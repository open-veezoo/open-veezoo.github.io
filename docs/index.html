Knowledge Graph

What is a Knowledge Graph?#

A Knowledge Graph is a version-controlled semantic model of a dataset, which is used by Veezoo to understand your question and generate the correct SQL query that answers it.

It is represented as a set of files using a modelling language called VKL, the Veezoo Knowledge Language.

Now, first of all: Don't Panic!

We will explore together the essence of this Knowledge Graph layer-by-layer, so you can get quickly up and running.

The Knowledge Graph is divided into different layers. We will focus here on the two most important ones: knowledge-base and database layers.

Database Layer#

The database layer (db) is a very thin layer that contains exactly how the data is stored in the underlying database system.

It involves concepts like:

  • Schema
  • Tables
  • Columns
  • Primary and foreign keys

Here is an example of a table definition in VKL:

db.postgres {
database toy {
table ORDERS {
identifier: "ORDERS"
column amount {
hasPrimitiveType: "DOUBLE"
identifier: "amount"
}
column country {
hasPrimitiveType: "VARCHAR"
identifier: "country"
}
column customer {
hasPrimitiveType: "VARCHAR"
identifier: "customer"
}
column date {
hasPrimitiveType: "DATE"
identifier: "date"
}
column gender {
hasPrimitiveType: "VARCHAR"
identifier: "gender"
}
column movie {
hasPrimitiveType: "VARCHAR"
identifier: "movie"
}
}
}
}

Knowledge Base Layer#

The knowledge-base layer (kb) is where the magic happens. The idea behind the kb layer is to model your data semantically, i.e. as if you would be explaining to a person what your database is really about.

For this modelling, we have to introduce a new vocabulary:

  • Classes
  • Entities
  • Relations
  • Literal types

Let's see in detail what each of these things actually mean.

Classes and Entities#

If you come from a software background, you may be familiar with the idea of a class as defined in object-oriented programming literature. In some sense, we are making your data more object-oriented. So, a class can be defined as a category of things. The best way to understand it is to go over some examples.

A class could be something like:

  • Customer
  • Country
  • Company
  • Event
  • Lead_Status

Let's see a brief excerpt from a class VKL definition:

kb {
class Country {
name.en: "Country"
label.en: "Country"
refersTo: onto.Country
semanticLink: {
...
}
}
}

We will look into this code in more detail later.

Now, each of these classes may have instantiations of it. In programming, we call these "objects". In VKL, we call them entities. So following our example, we may have entities like:

  • entity Till_Haug of class Customer
  • entity Brazil of class Country
  • entity Veezoo_AG of class Company
  • entity Lost of class Lead_Status

But what about Event? For classes like Event, Transaction and Order, it makes sense usually to model it as a class without entities. Prefer to use classes with entities when there is a way to refer to an individual of this class, e.g. in a question.

note

In case you are wondering, we borrowed the term entity from the NLP problem known as Entity Linking.

Here is an entity definition in VKL:

kb {
class Country {
entity Brazil_5692 {
name.en: "Brazil"
label.en: "Brazil"
sqlId: "BR"
}
...
}
}

Literal Types#

Literal types are your usual data types like: numbers, dates, booleans and strings.

Relations#

A relation in VKL describes how classes and other classes or literal types relate to each other.

Example:

  • Till_Haug works at Veezoo_AG
  • JP_Monteiro was born on 30th of March 1993
  • Brazil has a population of 209'500'000

In other words, we have the following relations:

  • works_at is a relation that goes from Customer to Company
  • birthdate is a relation that goes from Customer to the literal type date
  • population is a relation that goes from Customer to the literal type integer
kb {
relation customer_Country_From {
from: kb.Customer
to: kb.Country
name.en: "from"
...
}
}

As you can see, it doesn't really matter whether the relation is defined as a column inside a table or as a foreign key or actually a transformation on top of another column. It is just a relation. This means that 2 completely distinct database schemas can have the same knowledge-base layout, if they represent the same information.

This allows you to decouple how your data is represented in the database (due to whatever reasons like performance, convenience, etc) to what it actually means in the real world. How cool is that!

Still, how do the concepts in the kb layer relate to the ones in the db layer? The answer is called Semantic Links.
